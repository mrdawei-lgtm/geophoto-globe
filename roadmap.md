# Roadmap

项目：`GeoPhoto Globe`

当前目录：`/opt/geophoto-globe-dev`

## 产品方向

`GeoPhoto Globe` 后续按“两层产品能力”推进：

1. 平台主线：地理多媒体展示平台  
   面向旅行创作者、摄影博主、自然观察组织、天文爱好者等，用地图 / 地球来组织照片与视频内容。
2. 叙事扩展线：故事与时间轴讲述系统  
   在已有媒体库和地点数据之上，增加章节、时间轴、脚本化讲述、路线回放等能力。

当前优先级以“平台主线”优先，“叙事扩展线”作为二期能力。

## Viewer 体验原则

viewer 端后续不按传统多页面内容站来设计，而按“常驻 globe 的连续体验”来设计。

核心原则：

- 地球始终作为 viewer 主舞台存在
- 用户在项目、合集、媒体、故事等层级之间切换时，不卸载 globe，不刷新整个 viewer
- 所有层级通过 overlay、镜头转场、scope 切换来完成
- URL 仍然存在，但只驱动同一个 viewer shell 的状态，不切换成完全不同的站点形态

也就是说，后续更适合使用这些概念：

- 全局浏览视图
- 项目聚焦视图
- 合集聚焦视图
- 媒体聚焦视图
- 故事播放视图

它们共享一个常驻 `GlobeShell`。

## 产品定位

目标产品不是单纯相册，也不是纯叙事播放器，而是：

一个以地理位置为核心的多媒体展示与叙事平台。

它应同时支持：

- 按地点展示照片 / 视频合集
- 按项目、专题、作者、组织来管理内容
- 在媒体库之上继续构建时间轴、故事线、章节化讲述
- 在所有主要浏览层级中保持地球为核心视觉与交互舞台

## 阶段划分

### Phase 1: 从单管理员工具升级为可运营的平台

目标：把当前“单管理员照片 CMS + 公开地球页”升级为可支持多创作者、多项目、多内容状态的平台底座。

重点：

- 用户注册、登录、会话管理
- 角色权限体系
- 项目 / 合集 / 媒体的内容结构
- 内容审核与发布状态流转
- `GlobeShell` 连续 viewer 架构
- 项目级访问控制
- 项目级皮肤配置
- 多语言基础设施

### Phase 2: 多媒体内容能力升级

目标：让平台不再局限于图片，支持更完整的展示型内容生产。

重点：

- 视频上传、转码、封面、时长与播放支持
- 更完善的标签、专题、筛选和搜索
- 作者页、组织页、项目聚焦视图、合集聚焦视图
- 公开可见性控制，例如公开 / 未列出 / 私有 / 组织内
- 皮肤设计器与皮肤市场首版

### Phase 3: 故事与时间轴能力

目标：在平台内容底座上增加“讲故事”的能力。

重点：

- 时间轴浏览
- 故事脚本 / 章节编辑
- 地图与时间联动
- 路线回放
- 一次旅行、一个历史事件、一个文化演化主题的可视化讲述
- 故事播放过程保持 globe 常驻，时间轴和章节驱动镜头与 overlay 转场

## 目标用户

优先服务以下内容生产者：

- 全球旅行博主
- 自然观察 / 动植物记录组织
- 天文爱好者与观测社群
- 历史、文化、地理主题内容创作者

这些用户的共同需求是：

- 内容天然带有地点属性
- 需要比普通相册更强的组织方式
- 希望公开展示效果更有空间感和专题感
- 不希望地球只是入口，而希望它成为主要浏览方式
- 后续可能需要故事化表达

## 核心产品能力

### 1. 地理展示能力

- 继续保留 3D 地球作为标志性入口
- 地球不只作为入口，也作为 viewer 的持续主舞台
- 支持从全局、项目、合集、媒体、故事等不同 scope 进入地球视图
- 支持地点聚合、时间筛选、主题筛选
- 允许同一地点挂载多张照片和视频

### 2. 内容管理能力

- 每个账号或组织可拥有多个项目
- 每个项目下可管理多个合集、地点组和媒体资源
- 管理后台支持草稿、审核、发布、下线等状态
- 管理后台支持批量导入、批量编辑、批量审核
- 每个项目可配置访问级别，例如公开、密码访问、私有
- 每个项目可绑定默认皮肤、默认语言和 viewer 偏好

### 3. Viewer 定制能力

- 项目可绑定专属 viewer 皮肤
- viewer 皮肤可控制文字、面板、背景、地球、边境线、海岸线、聚合点等视觉元素
- 后续支持皮肤设计器与皮肤市场
- viewer 支持无刷新切换语言和内容文案回退

### 4. 叙事能力

- 一个故事可引用多个地点、媒体组和时间节点
- 一个故事拆分为多个章节或场景
- 每个章节可配置标题、说明、媒体、地点、时间和顺序
- 前台可按章节顺序进行浏览或播放

## 数据模型演进方向

当前代码已具备：

- `photos`
- `photo_groups`
- 导入任务
- 地理摘要
- 共享简介

后续 viewer 层建议引入正式的项目级配置能力，例如：

- 访问级别
- 访问密码
- 默认皮肤
- 默认语言
- 支持语言列表
- viewer 模式偏好

后续建议增加以下核心实体：

- `users`
- `organizations`
- `memberships`
- `projects`
- `collections`
- `media_items`
- `media_groups`
- `stories`
- `story_scenes`
- `skins`
- `skin_versions`
- `skin_market_entries`
- `moderation_logs`

演进原则：

- 媒体库是底座
- 地点分组继续保留
- 故事内容引用媒体与地点，而不是替代媒体模型
- 权限、审核、发布状态应成为一等能力
- viewer 配置应按项目或内容作用域加载，不应写死为全站单一主题

## 账号与权限路线

当前系统仍是单管理员密码模式，后续要升级为正式平台账号体系。

建议角色至少包括：

- `viewer`
- `creator`
- `editor`
- `reviewer`
- `admin`

建议内容状态至少包括：

- `draft`
- `in_review`
- `published`
- `rejected`
- `archived`

建议访问级别至少包括：

- `public`
- `password_protected`
- `private`

## 审核与运营路线

如果平台面向多创作者或组织，审核能力应尽早纳入主线，而不是后补。

建议具备：

- 创作者提交审核
- 审核人查看待审核内容
- 审核意见记录
- 拒绝后修改并重新提交
- 发布与下线操作留痕

## 皮肤系统路线

皮肤系统不是简单主题切换，而是 viewer 的正式视觉配置系统。

建议分三层推进：

### 1. 项目级皮肤绑定

- 每个项目可选择一个默认皮肤
- viewer 进入项目时自动加载该皮肤
- 皮肤同时作用于 globe 和 overlay

### 2. 皮肤设计器

- 提供管理端 skin editor
- 支持实时预览 globe 与 overlay 效果
- 支持保存草稿、复制已有皮肤、导入导出配置

### 3. 皮肤市场

- 支持浏览、预览、安装皮肤
- 支持把自定义皮肤发布到市场
- 支持其他项目复用市场皮肤
- 首版应保留审核与发布控制，避免无序增长

建议皮肤控制项至少包括：

- 字号与排版
- 文本颜色
- 面板颜色、边框、透明度
- 背景图
- 地球海洋 / 陆地配色
- 边境线 / 海岸线颜色
- 聚合点颜色
- 路线、高亮和媒体边框样式

## 多语言路线

多语言需同时覆盖两部分：

### 1. UI 多语言

- 系统界面文案
- viewer 控件文案
- 管理端操作文案

### 2. 内容多语言

- 项目标题与简介
- 合集标题与简介
- 媒体标题与描述
- 故事标题、章节与说明
- 皮肤名称与皮肤描述
- 市场展示文案

建议语言解析优先级：

- 用户手动选择语言
- 项目默认语言
- 浏览器语言
- 系统默认语言

多语言切换不应刷新 viewer，只更新当前 shell 中的文案和需要重渲染的标签。

## 媒体能力路线

当前项目以照片为主，后续应演进为“照片 + 视频”并存。

图片方向继续优化：

- 导入质量
- 封面策略
- 缩略图与展示图生成
- 批量编辑效率

视频方向新增能力：

- 上传
- 转码
- 封面帧
- 时长 / 分辨率 / 文件大小元数据
- 公开页播放体验

## Viewer 信息架构建议

后续 viewer 不应走“globe 消失、切到纯内容页”的路径，而应形成一个常驻 shell：

- `global` scope：全局浏览
- `project` scope：项目聚焦
- `collection` scope：合集聚焦
- `media` scope：媒体聚焦
- `story` scope：故事播放

每个 scope 都通过同一个 `GlobeShell` 承载：

- 地球保持常驻
- overlay 和面板切换层级信息
- URL 驱动 shell 状态
- 镜头和数据范围按 scope 平滑转场

## 技术路线建议

为支持平台化，后续建议：

- 鉴权从单密码升级到正式用户体系
- 数据库从“适合单机”的设计逐步朝平台化模型演进
- 媒体存储抽象为可切换本地磁盘 / 对象存储
- 审核、发布、操作日志进入正式数据模型
- 视频处理链路独立设计，不直接复用照片导入逻辑
- viewer 前端按 shell + scope + overlay 模型重构
- 皮肤 token 同时控制 Web UI 和 Three scene 视觉参数
- 多语言能力同时覆盖 UI 层与内容层

## 当前不作为主线的方向

以下方向可以保留为后续选项，但不应抢占当前主线优先级：

- 微信小程序管理员端
- 多端同时维护的专用后台
- 过早扩展复杂社交互动
- 过早支持多人实时协作编辑

这些方向不是不做，而是在平台核心模型、审核流、viewer 架构和媒体能力稳定后再评估。
