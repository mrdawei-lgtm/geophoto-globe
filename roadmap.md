## 微信小程序管理员端

### 目标

新增一个微信小程序管理员端，面向管理员使用。小程序支持从 iPhone 相册上传新照片、自动识别 EXIF / GPS、无 GPS 时手动地图补点、管理现有照片，以及替换单张原图并重生成衍生图。

小程序使用一套对微信环境友好的专用后端，但与现有 Web 项目共享同一份图库数据和资源存储，保证两端看到的是同一批照片。

### 范围

- 支持管理员登录
- 支持照片列表、搜索、筛选
- 支持单图详情编辑：
  - 标题
  - 描述
  - 地点名
  - 显示 / 隐藏
  - 删除 / 恢复
  - GPS 补点
- 支持从 iPhone 相册上传新照片
- 支持服务端自动解析 EXIF / GPS
- 支持无 GPS 照片在小程序内地图选点补 GPS
- 支持替换已有记录的单张原图
- 支持替换后重生成托管图、缩略图、展示图
- 支持替换后重新解析 EXIF / GPS

### 架构决策

- 小程序为管理员工具，不做公开访客浏览
- 小程序走专用后端，部署到外网 VPS
- 现有 Web 项目继续保留
- 两套后端共享同一图库数据和资源目录
- 继续沿用现有 `PhotoRecord` 作为核心数据模型
- 继续沿用现有存储结构：
  - `storage/originals`
  - `storage/managed`
  - `storage/thumbs`
  - `storage/display`
- 小程序鉴权使用固定管理员密码换取 token
- GPS 自动识别以服务端上传后解析 EXIF 为准
- 无 GPS 时，上传成功但标记为待补点，不直接公开展示
- GPS 补点交互固定为“地图选点 + 可编辑地点名”
- 图片替换固定为“替换单张原图并重建衍生资源”

### 接口规划

- `POST /miniapp/admin/login`
- `GET /miniapp/photos`
- `GET /miniapp/photos/:id`
- `POST /miniapp/photos/upload`
- `POST /miniapp/photos/:id/replace`
- `PATCH /miniapp/photos/:id`
- `POST /miniapp/photos/:id/gps`

### 行为约束

- 上传原图后由服务端统一生成托管图、缩略图、展示图
- 上传成功且有 GPS 的照片可直接进入可展示状态
- 上传成功但无 GPS 的照片进入待补点状态
- 补点后同步更新记录，并写回托管副本 EXIF
- 替换图片时保留原记录 `id` 和业务字段，只刷新资源文件和元数据
- 公开前台继续只显示：
  - `visible`
  - `deletedAt = null`
  - `hasGeo = true`

### 实施步骤

1. 抽取现有图片处理与数据存储逻辑，形成 Web 后端与小程序后端可共用的服务层。
2. 新建小程序专用后端，完成管理员鉴权、列表查询、单图编辑、上传、补 GPS、替换图片接口。
3. 新建微信小程序工程，完成登录页、照片列表页、照片详情页、上传流程、地图补点流程。
4. 打通小程序上传到共享图库存储的完整链路，并验证 EXIF / GPS 自动识别。
5. 打通单图替换链路，确保替换后衍生图与元数据同步重建。
6. 完成与现有 Web 管理端和公开前台的数据联通验证。

### 验收标准

- 管理员可以在小程序内登录并获取照片列表
- 可以从 iPhone 相册上传新照片
- 服务端可以自动识别带 GPS 的照片并直接入库
- 无 GPS 照片可以在小程序内地图补点
- 补点后照片可被公开前台展示
- 可以编辑现有照片的标题、描述、地点名、显示状态和删除状态
- 可以替换单张已有照片的原图，并正确重生成衍生资源
- 小程序与现有 Web 后台读写同一份数据，结果保持一致

### 风险与限制

- 当前系统只有 Web 管理后台，尚无适配微信环境的移动端管理入口
- 小程序端不应依赖前端直接读取 EXIF 作为唯一 GPS 来源
- 图片上传、替换、衍生图生成和 EXIF 写回都应由服务端统一处理
- 固定管理员密码适合首版，但不适合多管理员或更高安全要求场景
